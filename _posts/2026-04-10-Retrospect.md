## 주요 업무
- 데이터 관리
- 스테이지 이벤트 및 보상
- 보상용 패시브 스킬 구현
- 번역 작업


## Keep
- 작업하다 모르는 것이 있으면 직접 찾아가 물어보고, 기획자 - 나 - 개발자 이렇게 담당이 여럿인 기능은 둘을 오가며 최대한 기능 구현을 수정하지 않게함
- 데이터 테이블이 미구현된 상태에서 다른 개발자가 작업한 시스템을 분석하고, 데이터 테이블에 적용될 수 있도록 잘 연결함
- 후반부에 작업 일정이 잠시 비었을 때 우선순위가 낮아져서 미구현 상태로 출시할 예정이었던 다른 파트 작업을 구현함 


## Problem
- 데이터를 담은 틀은 빠르게 만들어졌는데 그 틀을 실제 제어 스크립트랑 연결하는 과정에서 시간이 많이 필요
- 첫 주차에 어떤 기획서가 나와 어떻게 개발할지 알 수 없어서 R&D를 하면서 시간을 너무 여유롭게 보냄
- 플레이어, 몬스터 등은 스텟과 스킬이 구현된 스크립트가 프리펩으로 되어있고, 블록은 SO에서 불러서, CSV 에서 데이터를 불러오는 방법이 통일되지 않고 둘을 불러오는 방법이 다름
- 로컬라이즈 기능을 만들었는데 언어 변경 시, UI 에선 실시간으로 변경이 되었으나 시나리오나 다른 대사에선 실시간으로 변경이 되지 않음
- 데이터 관리 기능은 중심 관리 매니저만 만들면 될 줄 알았는데, 실제로는 연결 기능까지 직접 작업해야되어서 기능 구현을 위한 일정 산정을 제대로 하지 못함
- 이벤트 보상이 간단할 줄 알고 클레스를 나누지 않고 플레이어 스탯 클래스에 추가했는데 생각보다 변수가 많아짐


## Try
- 기획서가 안 나와도, 예상해서 만들 수 있는 기능들을 미리 만들어봐야할 것 같음
- 데이터 테이블 에디터에서 SO로 만들고 런타임에서 데이터 매니저로 연결하는 기능 만들기
- 시나리오 등 다른 작업하는 팀원에게 미리 로컬라이즈를 위해 스크립트에서 텍스트를 생성하면 이벤트 기능과 언어 변경 시 갱신 기능을 넣을 수 있게 짜달라고 요청을 드려야 할 것 같음
- 플레이어와 플레이어 보상 클래스를 나누어, 시각적으로 더 관리하기 편하게 작업해야 할 것 같음

## Keep
- 각종 시각 자료와 제작 툴이 있어서 기획서를 이해하기 편했음
- 스크립트 상단이나 기획 문서 제목에 제작자 이름이 적혀있어, 작업을 하다 모르는 것이 생기면 작성자에게 직접 물어볼 수 있어서 고민하는 시간을 줄일 수 있었음


## Problem
- 데이터 테이블의 변수 종류와 명칭은 개발된 스크립트의 변수명과 달라서 작업을 할 때 혼선이 생기는 경우가 있었음
- 프리펩이나 스크립트 등이 각각의 작업 폴더에 있어서 폴더 경로를 따라가 원하는 파일을 찾기 어려웠음


## Try
- 스크립트 상단에 제작자, 이 스크립트의 역할이 간단히 적혀있는데, 추가로 어느 스크립트와 연결되는지도 적혀있다면 연결된 스크립트를 보면서 구조를 더 빠르게 파악할 수 있을 것 같음
- 데이터 테이블의 변수 종류와 명칭은 개발된 스크립트의 변수명과 통일해서 만드는 것이 추후 연결 작업을 할 때 혼선이 적을 것 같음
- 개발 중간(빌드 이전 디벨롭 브렌치 등)에 작업 폴더 정리하는 기간을 가져, 경로를 따라 쉽게 파일을 직관적으로 접근할 수 있도록 수정하면 해당 파일 찾는 수고를 덜 수 있을 것 같음
