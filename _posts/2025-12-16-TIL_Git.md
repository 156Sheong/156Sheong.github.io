## B. 버전의 정의
- 소프트웨어의 특정 시점 릴리즈를 나타내는 숫자와 영문자 식별자
- 업데이트, 버그 수정, 새로운 기능 추가 등 변화를 구분하고 추적하기 위해 사용


### Ba. 사용 단계에서 버전의 주요 역할
- **변경 이력 관리**
  - 소프트퉤어의 개발 과정 추적 및 어떤 변경 사항이 어느 시점에 적용되었는지 기록
- **기능 및 안정성 표시**
  - 버전 번호를 통해 큰 기능 변경(Major)과 새 기능 추가(Minor), 버그 수정(Ptach) 등 구분
- **호환성**
  - 버전 번호 변경으로 이전 버전과의 호환성 여부 예측 가능
- **사용자 식별**
  - 사용자가 자신의 앱 버전과, 업데이트 내역을 화인 가능


### Bb. 개발 단계에서 버전 관리(변경 이력 관리)가 필요한 이유
- **히스토리 추척 및 관리**
  - 작업물을 누가, 언제, 왜 수정했는지 등을 기록하여 변경 내용을 쉽게 확인 가능
  - 필요 시 특정 시점의 버전으로 롤백 가능
- **협업 효율성 증대**
  - 다른 사람의 작업 내역을 투명하게 공유하여 개인의 기여도 확인 가능
  - 여러 개발자가 동시에 작업해도 충돌 방지하고 충돌이 나도 쉽게 해결하도록 지원
- **백업 및 복구**
  - 프로젝트 파일 손상이나 오류 발생 시 안정적으로 복구하여 데이터 유실 위험 최소화
- **안정성 확보**
  - 새로운 기능 추가나 버그 수정 시 발생할 수 있는 위험 줄여서, 대규모 개발 과정을 안전하게 이끌 수 있음
- **자동화 촉진**
  - 버전이 저장될 때마다 테스트, 코드 분석, 배포 등의 작업을 자동화하여 프로세스 효율화 가능
- **작업 분리 및 병합**
  - 브랜치(Branch) 기능을 통해 새로운 기능 개발이나 실험할 때 메인 코드에 영향을 주지 않고 독립적으로 작업할 수 있음
  - 브랜치는 필요할 때 메인 코드에 병합(Merge)할 수 있음



## C. 버전 관리의 도구(VCS, Version Control System)
- 문서, 설계도, 소스 코드 등의 변경 사항을 관리해주는 소프트웨어
- 전문적인 버전 관리 프로그램이 존재
- 요즘은 Visual Studio 등 IDE에 버전 관리 프로그램이 통합되기도 함
- 구글 문서 도구, 원드라이브 등 클라우드 컴퓨팅에서도 버전 관리 기능 제공
  
### Ca. VCS의 방식에 따른 분류
- **로컬 VCS**
  - 서버 없이 로컬 컴퓨터 내에서 관리하는 형태
  - 단순하지만 협업이 힘들고, 컴퓨터의 정보가 날아가면 복구하기 사실상 불가능
- **중앙집중식 VCS**
  - 서버에 최종본이 있고, 사용자는 수정이 필요한 파일만 로컬로 받아 수정한 후 서버에 올리는 형태
  - 간단한 방법으로 협업 가능하고, 관리자가 누가 어떤 일을 하는지 알기 쉬움
  - 중앙 서버가 다운되면 그 기간은 업무가 마비, 서버 정보가 날아가면 모든 히스토리가 날아감
  - 대표 시스템 : `Subversion(SVN)`
- **분산 VCS**
  - 파일은 서버에 저장하지만, 수정을 위해 프로젝트 전체를 로컬 컴퓨터에 다운 받은 후 수정하는 형태
  - 중앙 서버가 다운되어도 각자 작업 가능, 서버가 날아가도 다운 받은 내용 남음
  - 최종적으로 병합(Merge)된 파일이 서버에 남음
  - 대표 시스템 : `Git`

### Cb. 주요 VCS
- **Git**
  - 현재 가장 보편적인 VCS
  - 분산 VCS
  - 파일 변경 사항을 매우 쉽고 명확하게 추적
  - Git Bash라는 Command Line(CLI) 유틸 제공
    - 문자열 형태로 명령어를 입력해 컴퓨터 시스템과 상호작용하고, 특정 작업을 수행하는데 사용하는 독립 프로그램
- **Subersion(SVN)**
  - 대표적인 중앙집중식 VCS
  - Git에 비해 사용하기 편함
  - 규모가 커질수록 충돌을 해결하기 위해 필요한 노력이 증가함
  - 파일 수정 시간, 서명된 수정사항을 저장하지 않음
- **Perforce Helix Core**
  - 중앙집중식 VCS
  - 바이너리 형태의 파일을 관리하는 데 장점이 있음
  - 대규모 확장 가능
  - 여러 작업 공간을 관리하는 것이 어려움



## D. Git 호스팅 웹 서비스
- Git 저장소 서버를 대신 유지 관리하는 서비스
- 대표적인 Git 호스팅 웹 서비스
  - **GitHub**
    - 대표적인 Git 호스팅 웹 서비스
    - 무료인 대신 자신의 소스 코드가 오픈됨
    - (GitHub 사용 예정)
  - **GitLab**
    - 자신의 서버에 설치하여 서버내 Private 저장소를 생성 가능
    - 기업이나 보안성을 중시하는 경우 사용
  - **Bitbucket**
    -  Jira, Confluence 등 Atlassian 제품군과의 연동 가능
    -  소규모 팀을 위한 Private 저장소 제공



## E. Git 기본 용어
### Ea. 주요 개념 용어
- **Repository(저장소)**
  - 프로젝트의 모든 버전 기록이 저장되는 공간
  - 로컬 저장소(내 컴퓨터)와 원격 저장소(리모트)가 있음
  - 히스토리, 태그, 소스의 브랜치(Branch)에 따라 버전을 저장, 작업자가 변경한 모든 히스토리 확인 가능
- **Working Directory**
  - 실제 파일을 수정하고 작업하는 공간
- **Staging Area**
  - 저장소에 커밋하기 전에 커밋을 준비하고, 커밋할 파일을 모아두는 단계
- **Clone**
  - 원격 저장소의 복사본을 로컬 저장소로 초기 복제하는 것
- **Commit**
  - 로컬에서 현재 변경된 작업을 마쳤을 시, 이를 확정하고 저장소에 저장하는 작업
  - 파일 추가/수정/삭제 작업은 **Working Directory** -> **Staging Area** -> **Commit** 순서로 이루어짐
- **Push**
  - 로컬 저장소의 커밋을 원격 저장소로 보내는 것
- **Pull**
  - 원격 저장소의 변경 내용을 로컬 저장소로 가져오는 것
- **Branch**
  - 특정 시점의 프로젝트 상태를 복제하여 독립적으로 개발할 수 있는 가지 공간
- **Head**
  - 현재 작업 중인 브랜치를 가리키는 포인터
- **Merge(병합)**
  - 다른 브랜치의 변경 내용을 현재 브랜치로 합치는 작업
- **Checkout**
  - 특정 시점이나 브랜치의 소스 코드를 이동하는 것


### Eb. 주요 명령어
- 명령 프롬프트, 터미널, 또는 Git Bash 등 CLI 도구를 사용하여 버전을 관리할 때 사용
- `git init`
  - 새로운 Git 저장소를 초기화
  - 이 명령어를 실행해야 추가적인 명령어를 사용하여 작업 진행 가능
- `git status`
  - 저장소의 현재 상태(변경된 파일, 커밋이 필요한 파일 등)를 확인
- `git clone` <서버 저장소 url>
  - 원격 저장소를 통째로 복제해서 로컬에 로컬 저장소로 가져옴
  - 원격 저장소의 모든 파일과 수정 이력을 모두 내려받음
- `git add` <파일/디렉토리 경로>
  - 변경 내용의 일부만 스테이진 영역에 추가
- `git add .`
  - 현재 디렉토리의 모든 변경 내용을 스테이징 영역에 추가
- `git add -A`
  - 작업 디렉토리 상 모든 변경 내용을 스테이징 영역에 추가
- `git add -p`
  - 각 변경 사항 중 스테이징 영역에 넘길 내용을 터미널에서 하나씩 확인하며 추가
- `git commit`
  - 스테이징 영역의 변경 사항을 커밋하여 로컬 저장소에 변경 사항을 확정하고 기록함
  - `git commit -m` <커밋 메세지> 의 형태를 통해 메세지와 함께 커밋
- `git push`
  - 변경 사항을 원격 저장소에 업로드
- `git pull`
  - 원격 저장소의 변경 내용을 로컬 저장소로 가져오고 자동 병합
- `git fetch`
  - 원격 저장소의 변경 내용을 로컬 저장소로 가져오지만 자동 병합하지 않음
- `git branch` <브랜치 명>
  - 새로운 브랜치를 생성
- `git merge` <브랜치 명>
  - 지정된 브랜치를 현재 브랜치롤 병합
  - 브랜치 A <- 브랜치 B의 형태로 한쪽이 합병됨
- `git checkout` <브랜치 명>
  - 다른 브랜치로 이동
- `git log`
  - 커밋 기록을 확인
